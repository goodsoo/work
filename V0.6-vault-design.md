# V0.6 — Vault 마이그레이션 design doc

옵시디안 방식 로컬 md 파일 vault로 백엔드 전환.

> **Status**: `/plan-eng-review` 통과 (2026-05-17). 주요 변경: VaultIndexer 제거, PWA 빌드 폐기, 모바일은 V0.7+ candidate, test plan 강화.

---

## 1. Goals & Non-goals

### Goals
- 모든 데이터를 사용자가 지정한 vault 폴더 안의 md 파일로 저장
- 옵시디안 (데스크탑/모바일) 으로 같은 vault 열어서 보기/편집 가능
- 회사 공유 시 회의록 1개 = 1개 md 파일 첨부로 self-contained
- iCloud/개인 GitHub 등 어디든 vault 폴더만 두면 동기화 가능
- 데스크탑 앱 UI/UX (3-탭 메모장, 캘린더, 통합 할 일 등)은 V0.5.3과 동일하게 유지
- Auth/서버 완전 제거

### Non-goals
- **모바일 native 앱** — 옵시디안 모바일로 대체. V0.7+ Tauri 2 Mobile candidate.
- **PWA/브라우저 빌드** — V0.6 부터 Tauri 데스크탑 전용. PWA 진입점/manifest/service worker 제거.
- **양방향 sync 서버** — vault 폴더 자체를 OS 동기화(iCloud, GitHub 등)에 위임
- **멀티 유저 / 권한**
- **기존 Supabase 데이터 마이그레이션** — 새로 시작
- **AI 자동 요약** — "Claude 프롬프트 복사" 헬퍼로 대체
- **옵시디안 Tasks 플러그인 호환 syntax** — 자체 `— 5/22` 형식만. emoji 형식 (`📅 2026-05-22`) 인식 안 함.

## 3. Vault 파일 schema

### 3.1 폴더 구조

```
<vault-root>/
├── inbox.md
├── meetings/
│   └── YYYY-MM-DD-{slug}.md
└── journals/
    └── YYYY-MM-DD.md
```

`schedules/` 폴더 없음 — 미래 일정도 inbox.md 또는 어디든 `- [ ]` 형태로 (옵시디안 native).

### 3.2 Meeting 파일

```markdown
---
title: 팀 주간 회의
date: 2026-05-16
time: "14:00"
attendees: [홍길동, 김철수]
tags: [meeting, team]
---

# 본문
회의 중 정리 노트…

# 회의 내용
(STT 결과)

# 요약
## 논의 사항
- …
## 결정 사항
- …
## 액션 아이템
- [ ] [홍길동] 보고서 — 5/22 #meeting
```

**파일명 규칙**: `YYYY-MM-DD-{slug}.md`. slug는 title을 그대로 사용 (한글 OK, 공백은 하이픈, 슬래시/콜론만 escape). 같은 날 같은 title 충돌 시 `-2`, `-3` suffix.

**H1 → 탭 매핑 convention**:
- `# 본문` → 본문 탭 (`content` 필드)
- `# 회의 내용` → 회의 내용 탭 (`transcript`)
- `# 요약` → 요약 탭. H2 3개 (`## 논의 사항`, `## 결정 사항`, `## 액션 아이템`)로 더 split → 기존 callout 3개에 매핑
- 매핑 안 되는 H1 → "미분류" 영역에 보존 (편집 가능, 손실 X)

**Frontmatter 필드**:
- `title` (string, required)
- `date` (YYYY-MM-DD, required)
- `time` (HH:MM string, optional)
- `attendees` (string[], optional)
- `tags` (string[], optional)
- 그 외 사용자가 추가한 필드는 그대로 보존하고 UI에서 무시

### 3.3 Journal 파일

```markdown
---
date: 2026-05-16
mood: 보통
---

본문 자유 형식 마크다운.

- [ ] 일기 안에서 발생한 todo도 vault 스캔에 잡힘
```

파일명: `YYYY-MM-DD.md`. 하루 한 파일 강제. 같은 날 두 번째 일기 작성 = 같은 파일에 append.

### 3.4 Inbox / 기타 파일

- `inbox.md` 는 root에 항상 존재 (없으면 첫 실행 시 생성)
- 그 외 사용자가 임의로 만든 `*.md` 파일은 vault 스캔 시 `- [ ]` 만 추출해서 통합 todo 뷰에 표출
- 옵시디안에서 만든 파일도 자동 인식

### 3.5 Todo / Event inline syntax (자체 형식, 옵시디안 Tasks 호환 X)

```
- [ ] [담당자] 할 일 내용 — 5/22 14:00 #tag1 #tag2
- [x] 완료된 항목
- [ ] 단순 todo (담당자/기한/태그 모두 optional)
```

파서 규칙:
- `- [ ]` / `- [x]` 체크박스
- `[이름]` 접두 → assignee (optional)
- `— M/D` 또는 `— YYYY-MM-DD` → due date (M/D는 현재 연도 가정)
- `— HH:MM` 또는 `— M/D HH:MM` → time (event 식별 힌트)
- `#tag` → tags (multiple)
- 나머지는 text

**Todo vs Event 구분**:
- `#event` 태그 있거나 시간(HH:MM) 명시 → event (캘린더에 시간으로 표시)
- 그 외 → todo (캘린더에 종일 표시)

**비결정**: 옵시디안 Tasks 플러그인 emoji 형식 (`📅 2026-05-22`) 은 인식 안 함. 옵시디안 모바일에서는 일반 체크박스로만 보임 (필터/정렬 X). 가독성 우선.

## 4. 아키텍처 레이어

```
┌─────────────────────────────────────────┐
│ UI (React components, V0.5.3 유지)        │
├─────────────────────────────────────────┤
│ Hooks (useMeetings/useJournals/useTodos │
│   /useSchedules — 인터페이스 유지)         │
├─────────────────────────────────────────┤
│ TanStack Query (cache, optimistic)       │
│  ↑ queryClient.invalidateQueries()       │
│  └ FileWatcher (외부 변경 감지 시 호출)    │
├─────────────────────────────────────────┤
│ Vault Repository (CRUD on vault)         │
│  ├─ MarkdownParser                       │
│  ├─ TaskExtractor                        │
│  └─ Scan helpers (list + extract todos)  │
├─────────────────────────────────────────┤
│ VaultAdapter (Tauri fs API 래퍼)          │
├─────────────────────────────────────────┤
│ Tauri plugins (fs / fs-watch / dialog)   │
└─────────────────────────────────────────┘
```

**핵심 원칙**:
- hooks 상위 인터페이스는 V0.5.3과 동일. UI는 거의 안 건드리고 백엔드만 swap.
- **별도 인덱서 모듈 없음**. TanStack Query 자체가 캐시 레이어 역할. file watcher 가 변경 감지 시 해당 query key invalidate → 다음 read 시 vault 재스캔.

**필요 Tauri plugin**:
- `@tauri-apps/plugin-fs` — fs CRUD
- `@tauri-apps/plugin-fs-watch` (또는 fs plugin 의 watch API) — 외부 변경 감지
- `@tauri-apps/plugin-dialog` — vault 폴더 picker

## 5. 핵심 모듈 시그니처

### 5.1 `VaultAdapter` (`src/lib/vault/adapter.ts`)

Tauri fs API 직접 호출 래퍼. 테스트용 in-memory 구현체로 대체 가능.

```typescript
export interface VaultAdapter {
  setRoot(path: string): Promise<void>;
  getRoot(): string | null;

  list(subdir: string): Promise<string[]>;          // 상대 경로 리스트
  read(path: string): Promise<string>;
  readMeta(path: string): Promise<{ mtime: number; size: number }>;
  write(path: string, content: string, expectedMtime?: number): Promise<void>;  // atomic + optimistic concurrency
  delete(path: string): Promise<void>;
  rename(from: string, to: string): Promise<void>;
  exists(path: string): Promise<boolean>;

  watch(callback: (event: VaultWatchEvent) => void): () => void;
}

export type VaultWatchEvent =
  | { type: 'created'; path: string }
  | { type: 'modified'; path: string }
  | { type: 'deleted'; path: string }
  | { type: 'renamed'; from: string; to: string };
```

**Atomic write**: tmp 파일에 쓰기 → fsync → rename. 중간 crash 시에도 원본 안 깨짐.

**Optimistic concurrency**: `write()` 가 `expectedMtime` 받음. 디스크 현재 mtime 이 그와 다르면 throw — 호출자가 §7.1 conflict resolution 처리.

**Watch**: `tauri-plugin-fs-watch`. macOS FSEvents 기반. debounce 100ms.

### 5.2 `MarkdownParser` (`src/lib/vault/parser.ts`)

```typescript
export interface VaultFile {
  raw: string;
  frontmatter: Record<string, unknown>;
  sections: Map<string, string>;       // h1 title → body (heading 제외)
  unmapped: string;                     // 매핑 안 된 H1 섹션 raw
}

export function parseVaultFile(raw: string): VaultFile;
export function serializeVaultFile(file: VaultFile): string;

// 부분 patch (성능: 전체 직렬화 안 함, 텍스트 replace)
export function patchFrontmatter(raw: string, updates: Record<string, unknown>): string;
export function patchSection(raw: string, h1Title: string, newBody: string): string;
```

**파싱 디테일**:
- Frontmatter: `gray-matter` (Layer 1, mature) 사용. 자체 구현 안 함.
- H1 split: `^# (.+)$` regex, 다음 H1 직전까지가 body. 코드블록(```) 안 H1 무시.
- Section body는 H1 헤더 라인 제외하고 trim
- Unknown H1 (`# 본문`/`# 회의 내용`/`# 요약` 외) → `unmapped`에 raw 보존
- 빈 섹션 = `sections.get('본문') === ''`

### 5.3 `TaskExtractor` (`src/lib/vault/tasks.ts`)

```typescript
export interface TodoItem {
  id: string;                // 파일경로 + 라인 콘텐츠 hash
  text: string;
  done: boolean;
  due?: string;              // ISO
  time?: string;             // HH:MM
  assignee?: string;
  tags: string[];
  source: { file: string; line: number };
  isEvent: boolean;          // #event 태그 or time 있음
}

export function extractTodos(filePath: string, raw: string): TodoItem[];

// 체크박스 토글: 정확한 라인의 [ ] ↔ [x] 만 patch
export function toggleTodo(raw: string, line: number, done: boolean): string;
```

**ID 안정성**: `${filePath}:${normalizedText}` 의 hash. 체크박스 토글은 텍스트 안 변하니 같은 ID 유지. 텍스트 수정 = 새 ID (사실상 새 todo). TanStack Query cache 는 query key 단위 invalidate 라 ID 변화에 무관.

**Line tracking**: `source.line` 으로 toggle 시 정확한 라인 위치 patch. 같은 텍스트의 todo 가 여러 줄에 있어도 line 으로 구분.

### 5.4 Hooks 시그니처 (V0.5.3 유지) + 캐시 동작

```typescript
// 인터페이스 그대로
useMeetings(): UseQueryResult<MeetingMeta[]>;       // 리스트 — frontmatter + 경로만
useMeeting(id: string): UseQueryResult<Meeting>;    // 본문 포함 — 메모장 열 때만
useCreateMeeting(): UseMutationResult<...>;
useUpdateMeeting(): UseMutationResult<...>;
useDeleteMeeting(): UseMutationResult<...>;
```

**캐시 전략 (Indexer 대체)**:
- `useMeetings()` query function = `vault.scan('meetings/')` + 각 파일 frontmatter parse. TanStack Query 가 결과를 staleTime (예: 60s) 동안 캐시.
- `useMeeting(id)` 별도 query — 본문 포함, 메모장 열릴 때만 fetch.
- `useTodos()` query function = vault 전체 `*.md` 스캔 + `extractTodos` 적용. 결과 캐시.
- File watcher 가 modified/created/deleted 이벤트 발행 시 → 해당 파일 경로의 query key 들을 `queryClient.invalidateQueries()` → 다음 read 시 vault 재스캔.

**Mutation 패턴**:
- `id` = 파일 경로 (`meetings/2026-05-16-팀-주간회의.md`)
- `useUpdateMeeting` optimistic: Query cache 업데이트 → `patchSection` + atomic write (with expectedMtime) → 성공 시 invalidate / 실패 시 rollback
- `useStateHistory` (4-stack: body/transcript/summary/meta) 변경 X — onCommit 만 fs write 로 swap

**성능 안전판**: 수백 파일 vault 스캔 < 50ms 예상. 실측 후 너무 느리면 frontmatter only 부분 캐시 추가 (별도 Indexer 모듈 없이 query function 안에서 처리).

## 6. 주요 시나리오

### 6.1 첫 실행

1. localStorage `vaultRoot` 비어 있음 → vault picker 모달
2. Tauri `dialog.open({ directory: true })` 로 폴더 선택
3. 선택한 폴더에 `inbox.md` / `meetings/` / `journals/` 없으면 생성
4. localStorage 저장 → 메인 UI 진입 (`useMeetings()` 등이 첫 스캔)

### 6.2 회의록 작성

1. `useCreateMeeting({ title, date, time, attendees })` 호출
2. 파일명 생성: `meetings/${date}-${slug(title)}.md`. 충돌 시 `-2` suffix
3. 빈 템플릿으로 파일 write (frontmatter + 빈 본문/회의 내용/요약 H1)
4. `queryClient.invalidateQueries(['meetings'])` → UI 자동 갱신

### 6.3 회의록 편집 (탭 별)

1. 본문 탭 textarea 편집 → useStateHistory commit
2. `useUpdateMeeting({ id: filePath, updates: { content } })`
3. `patchSection(raw, '본문', newContent)` 로 부분 patch
4. atomic write (with expectedMtime — 외부 변경 없으면 진행)
5. file watcher 가 자기 변경 감지 (자기 write 직후 modified 이벤트 발행 → 가장 최근 자기 write mtime 과 일치하면 skip)

### 6.4 외부 도구로 vault 편집

1. 옵시디안 모바일에서 회의록 편집 → iCloud sync → 데스크탑 파일 mtime 변경
2. file watcher 가 `modified` 이벤트 발행
3. `queryClient.invalidateQueries({ predicate: q => q.queryKey 가 해당 파일 포함 })` 호출
4. 다음 read 시 vault 재스캔 → UI 자동 reload
5. **현재 열려 있는 메모가 외부에서 바뀜**: editing 중이면 충돌 가능 → §7.1 참조

### 6.5 통합 할 일 뷰

1. `useTodos()` 호출 시 vault 전체 `*.md` 스캔 + 각 파일 `extractTodos` → TanStack Query 캐시
2. 캘린더 뷰: due date 있는 항목만 표시. event (시간 있음/`#event`) 와 todo 구분 표시
3. 할 일 페이지: 전체 todo. tag 필터 (#work / #meeting / #personal / 미분류)
4. 체크박스 toggle → `toggleTodo(raw, line, done)` 으로 해당 파일 라인만 patch → atomic write → invalidate

### 6.6 "Claude 프롬프트 복사"

1. 메모장 우측 액션에 새 버튼 추가
2. 클릭 → 본문 + 회의 내용을 묶어 프롬프트 생성:
   ```
   다음 회의록을 정리해주세요.
   - 논의 사항
   - 결정 사항
   - 액션 아이템 ([담당자] 할 일 — 기한 형식)

   ## 본문
   {content}

   ## 회의 내용
   {transcript}
   ```
3. clipboard 복사 + toast
4. (V0.7+ 후보) Claude 응답 paste 시 H2 split → 요약 탭 callout 자동 채움. **V0.6 에서는 수동 paste**.

## 7. Edge cases & 견고성

### 7.1 동시 편집 충돌
**시나리오**: 옵시디안 모바일이 같은 파일을 수정한 사이에 데스크탑 앱도 수정.

**전략**:
- 메모장 열 때 `readMeta()` 로 mtime 저장 (= `expectedMtime`)
- `write()` 호출 시 디스크 현재 mtime 과 `expectedMtime` 비교 (`VaultAdapter` 단에서 처리)
- 다르면 throw `ConflictError` → 호출자(mutation)가 UI 처리
- UI 에 "외부에서 변경됨" 모달 + "내 변경 보존하기" / "외부 변경 가져오기" 선택지
- "보존" 선택 시: 외부 버전을 `.{원본}.conflict-YYYYMMDD-HHMMSS.md` 로 보존 + 우리 변경 정상 write
- conflict 파일은 우리 자체 형식 (iCloud sync 의 conflict 파일 `(conflicted copy)` 와 구분)

### 7.2 사용자가 H1 헤더 변경
**시나리오**: 옵시디안에서 `# 본문` → `# 회의록` 으로 rename.

**전략**:
- 파서가 알려진 H1 매핑에 실패 → `unmapped` 에 보존
- UI 에서 "미분류 섹션" 영역으로 보여줌 (편집/삭제 가능)
- 본문 탭은 빈 상태로 보임 (사용자가 의도적으로 한 거면 OK)
- 데이터 손실 절대 X

### 7.3 손상된 frontmatter
**시나리오**: YAML 문법 오류.

**전략**:
- `gray-matter` parse error catch
- frontmatter 빈 객체로 fallback, raw 보존
- UI 에 경고 배지 ("frontmatter 파싱 실패, 메타데이터 입력 시 새로 덮어쓰여집니다")

### 7.4 파일명 충돌
**시나리오**: 같은 날 같은 title 회의 2개.

**전략**: `-2`, `-3` 자동 suffix. `2026-05-16-팀회의-2.md`.

### 7.5 vault 폴더 사라짐/권한 없음
**시나리오**: 사용자가 외장 디스크 vault 두고 disconnect.

**전략**:
- adapter 호출 시 fs error → "vault 접근 불가" 모달
- 다시 picker / 재시도 / 종료 선택지

### 7.6 큰 transcript 성능
**시나리오**: 1시간 회의 STT = ~30KB ~ ~100KB.

**전략**:
- 본문은 메모장 열 때만 read (lazy, `useMeeting(id)` 트리거)
- transcript 탭 textarea 렌더링은 React 가 처리 (수만 줄 textarea 도 무난)
- todo 스캔은 정규식 line 단위로 빠르게 — 본문 전체를 react-markdown 으로 파싱하지 않음

### 7.7 한국어 파일명
**Tauri fs API**: UTF-8 native. macOS HFS+/APFS, iCloud, GitHub 모두 정상. 슬래시(`/`), 콜론(`:`) 만 escape.

### 7.8 atomic write 실패
**시나리오**: tmp write 까진 됐는데 rename 실패.

**전략**:
- tmp 파일은 `.{원본}.tmp` 이름. cleanup 잡 (앱 시작 시 `.tmp` 파일 발견하면 삭제)
- write 함수는 throw — 호출자(mutation)가 toast 표시 + 사용자 재시도

### 7.9 옵시디안/시스템이 만드는 메타 파일
**시나리오**: `.obsidian/`, `.trash/` 폴더, `.DS_Store` 등.

**전략**:
- vault 스캔 시 `.` 시작 폴더/파일 무시
- root level 파일 중 `*.md` 만 인식
