# SIMPLIFY — 앱 단순화 기획 (초안)

작성: 2026-06-14. 상태: 착수 1 완료 (PR #78, "오늘" 대시보드) + 캘린더 폐기 완료 (PR #79). 모델 분리(2번) 등 나머지 기획 단계. 이 문서는 살아있는 초안이라 결정 바뀌면 갱신.

## 왜

기능은 충분히 만들었는데 너무 많아 보이고 복잡해서 손이 잘 안 감. "오늘 나에게 중요한 것"을 보여주는 화면도 없음. 추가가 아니라 편집(없앨 건 없애고 단순화)이 필요한 시점.

## 진단: 핵심은 노트+할 일인데, 무게는 안 쓰는 기능에 쏠림

매일 실제로 쓰는 건 메모장(노트) + 할 일. 그런데:

- 앱을 켜면 안 쓰는 캘린더 그리드로 떨어짐.
- 액티비티 바 1/4을 매일 안 쓰는 포트폴리오(가장 무거움, 4,200줄)가 차지.
- "오늘 뭐가 중요한가"가 어디에도 안 모임.

복잡도가 정작 매일 쓰는 노트, 할 일이 아니라 가끔 쓰는 기능에 집중돼 있는 게 "손이 안 가는" 원인.

## 시간축 객체 모델 (앱의 척추)

카테고리 축을 없애고, 객체 종류 자체를 축으로. 4개 객체가 척추.

```
객체        모양                          완료      반복   저장              집
──────────────────────────────────────────────────────────────────────────────────
할 일       텍스트 + 날짜(선택) + 프로젝트   체크 1회   X      tasks/{프로젝트}.md  할 일 리스트 + 오늘
일정        시작/종료/시간 + 텍스트         없음      X      schedule.md        캘린더 + 오늘
루틴        텍스트 + 주기                  주기 체크  O      (기존)             오늘(체크) + 관리 뷰
노트        (기존 그대로)                 ―         ―      notes/             메모장
```

### 할 일 (Task)

- 필드: 텍스트 + 날짜(선택) + 프로젝트.
- 완료 체크 있음 (일회성).
- 날짜 선택: "언젠가 할 일"(날짜 없는 inbox) 허용. 날짜 없으면 오늘/밀린 집계에서 빠지고 inbox에만 남음.
- **카테고리(work/schedule/other) 필드 통째 제거.** 경직된 고정 카테고리를 버리고 실제 일하는 단위인 **프로젝트 축**으로 교체 (복잡도 추가가 아니라 맞는 축으로 교체).
- **프로젝트 = 파일.** `tasks/` 폴더에 프로젝트당 md 파일 하나:
  ```
  tasks/
    inbox.md          ← 미분류 / 빠른 할 일
    프로젝트A.md
    프로젝트B.md
  ```
  - 프로젝트 추가 = 새 파일, 삭제 = 파일 휴지통. (사용자가 원한 "쉽게 추가/삭제".)
  - 할 일 한 줄 포맷은 그대로 (`- [ ] 텍스트 --- 날짜`), 카테고리 태그만 제거. **프로젝트는 "어느 파일에 있나"로 표현**하고 라인엔 안 박음.
  - 할 일 탭 사이드바 = `tasks/` 폴더 리스팅 = 프로젝트 목록 (없앤 카테고리 필터 자리를 대체).
  - "오늘"/"밀린" 집계는 `tasks/` 전체를 가로질러 스캔 + 각 할 일에 프로젝트 칩(=파일명) 표시 → flat 뷰에서도 구분.
  - 프로젝트 간 이동 = 라인을 파일 간 이동 (노트 MoveFolderModal 패턴 재사용 가능).
  - `scanAllTasks`가 지금 `inbox.md` 한 파일만 읽음 → `tasks/` 폴더 전체 스캔으로 변경. TaskItem에 이미 `source.file` 있어 프로젝트가 자연히 따라옴.
- **포트폴리오 프로젝트와 이름 정렬**: 포트폴리오에 이미 `portfolio/projects.md`(프로젝트 그룹)가 있음. task 프로젝트를 `tasks/`로 **독립**으로 두되 이름만 맞춤 → 나중에 "프로젝트 = 진행 중 할 일 + 출시한 작업" 통합 뷰가 싸짐. 지금 두 목록을 한 source로 합치진 않음 (over-engineering 회피).

### 일정 (Schedule)

- 필드: 시작일 + 종료일 + 시간 + 텍스트.
- **체크박스 없음** (이벤트는 "완료"하는 게 아니라 "발생/참석"하는 것).
- **별도 파일 `schedule.md`** (vault 루트, `tasks/`와 나란히). 체크박스 없는 날짜-우선 마크다운 리스트:
  ```
  - 2026-06-15 14:30 팀 회의
  - 2026-06-15 워크샵
  - 2026-06-15..2026-06-17 출장
  - 2026-06-15..2026-06-17 14:30 컨퍼런스
  ```
  포맷: `- {시작일}[..{종료일}] [{HH:MM}] {텍스트}`. 날짜 우선(일정은 "언제 뭐가"라 자연스럽고 정렬됨, task는 반대로 텍스트 우선 유지).
- 날짜/`..`범위/시간 토큰은 **기존 파서 정규식 재사용** (`ISO_DATE_RE`/`DATE_RANGE_RE`/`TIME_RE`). 체크박스 없는 leading-date 불릿 매칭하는 `extractEvents()` 작은 함수 하나만 신규.
- 옵시디안 Tasks 플러그인 호환이 제약 아니라 체크박스를 자유롭게 버릴 수 있음 (옵시디안 호환 방침 폐기 효과).

### 루틴 (Routine)

- 실사용 중. 제거 X.
- task와 다른 모양: 매 주기 반복 + "오늘 했나?" 로그/연속 기록이 핵심.
- 할 일 리스트에 섞지 않음 → 할 일 리스트를 순수 일회성만 유지 (단순화).
- 단, 무거운 1군 탭으로 만들지 않음:
  - 매일 동작(체크) → "오늘" 대시보드의 "오늘의 루틴" 블록.
  - 정의/연속기록 관리 → 가볍고 작은 별도 뷰 (가끔 들어가는 곳).

## "오늘" 대시보드 (신설, 기본 진입)

앱을 켜면 캘린더 그리드 대신 "오늘"로 진입. 체감 복잡도를 가장 크게 낮추는 한 방.

세로 블록 (군더더기 없이):

1. **오늘 일정** — 오늘 날짜의 일정을 리스트로 (그리드 아님), 시간순. 비면 "오늘 일정이 없습니다."
2. **오늘·밀린 할 일** — 오늘 마감 + 지난 미완료(밀린 건 빨강). 인라인 체크로 바로 완료.
3. **오늘의 루틴** — 오늘 체크할 루틴 목록.
4. **이어서 쓸 노트** (작게) — 최근 수정 노트 3~5개. 노트가 1순위 도구라 진입 동선.

상단에 노트/할 일 빠른 추가 한 줄. 위젯 더 안 늘림.

객체 모델과 1:1로 맞물림: 일정→블록1, 할 일→블록2, 루틴→블록3, 노트→블록4.

## 네비게이션 재배치 (3+3, 무게로 분리)

```
상단 (매일, 크게)            하단 (가끔, 작게)
  ● 오늘   ← 신설, 기본 진입    ○ 캘린더
  ● 노트                       ○ 내 작업(포트폴리오)
  ● 할 일                      ○ 설정
```

- 캘린더, 포트폴리오는 삭제 아님, 강등. 늘 보이는 1군에서 빼서 시선 경쟁 제거.
- 루틴 관리 뷰는 할 일 근처의 가벼운 진입점 (1군 탭 아님).

## 자르기 / 강등 / 합치기

자르기:
- 할 일 카테고리 필터 (전체/업무/미팅/미분류).
- 할 일 행마다 붙은 Gcal 동기화 칩 (매일 쓰는 화면의 시각 노이즈). Gcal 실사용이면 백그라운드로만, 안 쓰면 lib 14파일 제거 검토.

강등:
- 캘린더 그리드: 기본 진입에서 빼고, "특정 날짜 훑기" 용도로만. 오늘 일정은 대시보드가 담당.
- 포트폴리오: 유지하되 1군에서 강등 + 5초 자동 동기화 → 수동/진입 시 1회. (포트폴리오는 gh/claude CLI 의존이라 어차피 데스크탑 전용 강제.)

합치기:
- 휴지통 모달 6개 → 도메인 공용 1개.
- 설정 9섹션 → 핵심(Vault/표시) 위로, Backup/Attachments/Gcal은 "고급"으로 접기.

가볍게 두기:
- 일기: 그대로, "오늘" 화면에 한 줄만 노출.
- 스타일가이드(1,224줄): dev 전용이라 사용자 복잡도 0, 건드릴 필요 없음.

## 모바일: 앱을 나누지 말고 빌드 타깃별로 기능 표면만 나눈다

"두 개 앱"은 over-engineering. 데이터(iCloud vault)는 이미 하나라 나눌 게 없고, 포트폴리오는 CLI 의존이라 모바일에서 어차피 못 돎. → 하나의 Tauri 코드베이스에서 모바일 빌드는 가벼운 표면만 싣는다.

| | 모바일 빌드 | 데스크탑 빌드 |
|---|---|---|
| 오늘 대시보드 | ✅ (= 모바일 홈) | ✅ |
| 캘린더 | ✅ 읽기/가벼운 | ✅ 전체 |
| 간단 메모 | ✅ 읽기 + 빠른 편집 + 생성 | ― |
| 회의록 풀 에디터(음성/요약/AI) | ❌ | ✅ |
| 할 일 / 루틴 체크 | ✅ | ✅ |
| 포트폴리오 | ❌ (CLI 불가) | ✅ |

- 모바일에 실을 가벼운 표면 = 매일 쓴다고 고른 세트와 동일. **즉 데스크탑 단순화의 산출물이 곧 모바일 표면.** 별도 프로젝트 아님.
- 진짜 리스크는 UI 분리가 아니라 **iOS에서 iCloud Drive vault 파일 접근** (샌드박스 차이, iCloud 컨테이너 entitlement/document picker 필요 가능). Tauri 2 fs 플러그인이 iOS에서 vault 읽기/쓰기/watch 되는지가 make-or-break.
- 참고: 사용자는 옵시디안 앱을 안 씀 (2026-06-14). 따라서 "옵시디안 모바일로 갈음"은 모바일 fallback이 아니다. 폰에서 투두/일정/오늘을 보려면 전용 모바일 표면이 필요. (md 파일 vault + iCloud 동기화 자체는 유지.)

## 업무 vs 개인 — 분리하지 않음 (결정, 2026-06-15)

구현하다 보니 업무 전용 / 개인 전용 기능이 조금씩 생겨서 "앱을 업무용/개인용으로 분리할까" 고민이 들었음. 결론: **분리 안 함 (앱도 vault도).**

- 분리하려는 유일한 동기가 "기능이 갈라짐"이었음 (데이터 격리·집중·동료 공유 아님).
- 기능 갈라짐은 모든 앱의 정상 상태지 분리 근거가 아님. 쪼개면 공유 척추(오늘/노트/할일/일정/캘린더)를 복제·동기화해야 해서 비용만 늘어남.
- 이건 분리가 아니라 정돈(IA)으로 푸는 문제, 그 정돈은 이미 이 문서 계획에 들어있음:
  - 공유 척추는 그대로, 업무 전용(포트폴리오)은 2군 강등, 개인 전용(일기)은 가볍게 캘린더에 얹힘.
  - 업무/개인 구분은 이미 만든 grouping이 흡수: 할 일 = 프로젝트(`tasks/`), 노트 = 폴더(`notes/{폴더}/`). "개인: 운동"도 그냥 또 하나의 프로젝트.
- **탈출구(지금 만들지 말 것, YAGNI)**: 더 강한 동기가 생기면 한 단계만 올림. 명시 구분 필요 → 프로젝트/폴더 그룹화. 데이터 격리 필요 → vault 분리(코드 그대로). 동료 배포 필요 → 빌드 플래그(코드 그대로). 전부 한 코드베이스 위 점진 확장.

## 마이그레이션 메모

첫 배포(2026-05-27) 이후라 실데이터 있음 → lazy migration 필요 (첫 read 시 변환 + rewrite, V0.7.1 date-prefix→uid 패턴 선례).

`inbox.md` 한 번 읽어서 라인을 가른다:

- `#schedule` + 날짜 라인 → `schedule.md`로 이동 (체크박스 제거, `#schedule` 제거, 날짜 앞으로 재배치한 이벤트 포맷).
- 나머지 task(`#work`/`#other`/무태그) → `tasks/inbox.md`로 이동 (프로젝트 개념 없던 시절 거라 일단 전부 inbox). 체크박스 유지, 카테고리 태그만 제거. 이후 사용자가 프로젝트 파일로 재배치.

## 아직 안 정한 것

1. **루틴 관리 뷰의 정확한 위치/형태** (할 일 근처 진입점 형태).
2. **날짜 있는 task도 캘린더 그리드에 띄울지** — 저장 아니라 표시(UX) 결정, 나중에.
3. **프로젝트 순서/표시 이름 커스터마이즈** 필요 시 `tasks/projects.md` 인덱스 추가 검토 (portfolio/projects.md 미러). v1은 폴더 리스팅(이름순)으로 시작.
4. **모바일 추진 여부**: iOS vault 접근 스파이크 결과 보고 판단.

### 확정된 것 (참고)

- 일정 저장 = 별도 `schedule.md`, 체크박스 없는 날짜-우선 이벤트 포맷. (2026-06-15 확정)
- task 프로젝트 = `tasks/{프로젝트}.md` 파일 분리, inbox.md = 미분류. 포트폴리오 프로젝트와 이름 정렬(통합 X). (2026-06-15 확정)

## 추천 착수 순서

1. ✅ **"오늘" 홈 신설 + 기본 진입 변경.** (완료, PR #78) 위험 낮고 체감 가장 큼. 여기서 끝내도 절반 해결. 사이드바는 "다가오는 일정 아젠다"로 캘린더 그리드의 날짜 훑기 역할까지 일부 흡수 (메인은 오늘 고정). 일정 블록은 현재 모델 그대로 `category==="schedule"`로 표시 분리 — 실제 모델 분리는 2번에서.
2. **할 일/일정/루틴 모델 분리** + 카테고리 제거 + 일정 체크박스 제거 (+ lazy migration). 대시보드 블록과 직결.
3. ✅ **캘린더 강등 → 폐기** (완료, PR #79). 강등에서 더 나아가 캘린더 탭 자체를 없애고 날짜 훑기·일기 진입을 "오늘" 사이드바(1년 날짜 리스트)로 흡수. 남은 건 포트폴리오 자동 동기화/prominence 정리.
4. Gcal 칩 정리 + 휴지통/설정 통합.
5. (선택) iOS vault 접근 스파이크 → 되면 모바일 빌드에 가벼운 표면 반응형 노출.
